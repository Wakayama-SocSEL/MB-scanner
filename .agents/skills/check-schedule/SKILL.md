# check-schedule スキル

`TODO.md` を正本として、今日の日付と git log を突き合わせて進捗状況を報告する。

`check-research-alignment` が「**なぜ・どう設計するか (方針軸)**」を扱うのに対し、本スキルは「**いつ・どこまで (日付軸)**」に特化する。

## 起動時の前提確認

1. `pwd` で MB-scanner のプロジェクトルートにいることを確認
2. `TODO.md` の存在を確認。無ければ「スケジュールファイル (TODO.md) が見つからない」と明示して中断
3. 今日の日付はシステムコンテキスト `currentDate` から取得

## 質問パターンと対応手順

### A. 進捗状況 (例: 「今週何する？」「いまどこ？」「スケジュール通り？」)

1. `TODO.md` を Read
2. 今日の日付と「全体スケジュール」節の期間を照合し、**現フェーズ**を特定
3. 現フェーズの未チェック項目を抽出
4. 該当フェーズ期間の git log を取得:
   `git log --oneline --since="<フェーズ開始日>"`
5. コミットログ・PR マージから、**実体として終わっていそうな未チェック項目**を推定 (drift 候補)
6. 下記フォーマットで報告

### B. drift 検出 / 棚卸し (例: 「todo.md のチェックボックス合ってる？」「棚卸しして」)

1. `TODO.md` を Read
2. 直近の git log とマージ済みブランチを取得
3. 未チェックだが実体が完了していそうな項目を列挙 (根拠: commit hash / ファイル存在 / PR 番号)
4. 「更新候補」として列挙し、**ユーザー承認後に Edit で反映**

### C. 次週計画 (例: 「明日から何やる？」「次のフェーズは？」)

1. `TODO.md` の全体スケジュールから**次期間**を特定
2. そのフェーズの詳細節 (該当する場合) を抽出
3. 次期間のタスクリストを提示

## 出力フォーマット

### A の報告 (進捗)

```markdown
## 現在地
- 今日: <YYYY-MM-DD>
- 現フェーズ: <期間> <フェーズ名>
- 全体締切: <deadline>

## 今週の残タスク
- [ ] <未チェック項目> — 推定: 完了 / 未完 / 不明 (根拠)

## 直近の進展 (git log より)
- <commit hash> <件名>

## drift (未チェックだが実体完了の疑い)
- <項目> — 根拠: <commit / ファイル / PR>
```

### B の報告 (drift)

```markdown
## 更新候補
- [ ] → [x] <項目> — 根拠: <commit / ファイル / PR>
- ...

続けて `TODO.md` を更新しますか？
```

### C の報告 (次週計画)

```markdown
## 次フェーズ
- 期間: <start>〜<end>
- フェーズ名: <name>

## タスクリスト
- [ ] <項目>
- ...
```

## 編集ポリシー

- 読み取り・drift 検出は自動で行う
- `TODO.md` への書き込みは**必ずユーザー承認を得てから** Edit で実施
- チェックを外す (`[x]` → `[ ]`) 変更は drift ではなく再開の可能性があるため必ず確認
- スケジュール自体の書き換え (期間・フェーズの変更) は本スキルの対象外。ユーザーと直接相談

## 注意事項

- `TODO.md` は現状 230 行程度なので全読みで問題ない。肥大化したら節単位で Grep に切り替える
- git log は該当フェーズ期間に絞る (`--since`)。全履歴を取らない
- 方針・設計判断の質問は `check-research-alignment` に委譲する
- `current.md` 側の日付マイルストーンと食い違っていたら**両方正本化している警告**を出す

## 関連スキル

- `check-research-alignment`: 方針軸・設計判断 (*なぜ・どう*)
- `commit` / `reflect-changes`: フェーズ終了時に本スキルを呼んで todo.md 更新を提案する運用が望ましい
