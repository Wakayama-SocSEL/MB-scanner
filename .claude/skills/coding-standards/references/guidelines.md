# コーディング詳細ガイドライン

## 型定義の原則
本プロジェクトでは型安全性を最優先します。`Any` 型の使用は厳禁です。

### Pydantic vs TypedDict 選択基準
| ユースケース | 推奨する型 | 理由 |
| :--- | :--- | :--- |
| **外部入力の読み込み** | `Pydantic` | 厳密なバリデーションとパース機能が必要なため。 |
| **設定オブジェクト** | `Pydantic` | デフォルト値の管理や環境変数の読み込み(`pydantic-settings`)のため。 |
| **JSON出力** | `Pydantic` | `model_dump_json()` によるシリアライズとスキーマ生成を活用するため。 |
| **関数の内部戻り値** | `TypedDict` | ランタイムオーバーヘッドがなく軽量であるため。 |
| **内部データ構造** | `TypedDict` | バリデーション不要で、型ヒントの恩恵のみ受けたい場合。 |

### モデルの配置ルール
- **Pydanticモデル**: `mb_scanner/models/` ディレクトリに配置してください。
- **TypedDict**: 原則として使用するモジュール内に定義します。複数のモジュールで再利用する場合のみ `models/` へ移動してください。

## 外部ファイル形式の選択
データの永続化や出力形式を決定する際の基準です。

### JSON を選択すべき場合
- **ネスト構造**: データが階層構造を持つ場合（例: SARIF、抽出結果）。
- **型情報の保持**: 文字列、数値、真偽値、null を区別して保存したい場合。
- **Pydantic連携**: Pydanticモデルをそのままダンプしたい場合。

### CSV を選択すべき場合
- **分析用途**: Excelやスプレッドシートでの閲覧・分析が主目的の場合。
- **フラットデータ**: ネストのない単純なテーブル形式のデータ。
- **容量削減**: キー名の繰り返しを避け、ファイルサイズを抑えたい場合。

## JSONフォーマット処理
出力JSONの可読性向上のため、適切なフォーマット処理を行ってください。

### compact_json_array() の実装原則
- **対象**: プリミティブ値のみの配列（`[1, 2, 3]`, `["a", "b"]`）
- **除外**: オブジェクトを含む配列（`[{...}]`）、ネスト配列（`[[...]]`）
- **正規表現パターン**: `r"\[\s*(?:[^\[\]{}]*)\s*\]"`
  - `[^\[\]{}]` で `{`, `}`, `[`, `]` を含む配列を除外
- **検証**: エッジケース（プリミティブ、オブジェクト、ネスト）を網羅したテストを必ず作成
